## Part 0: What We're Building (The North Star)

A **public, interactive, single-page web application** that models your workplace through three lenses -- *Business Workflows*, *Systems & Infrastructure*, and *User Personas* -- and visualizes how they relate to each other. A personal "digital twin" of your work life that you can share in conversations, interviews, and with friends as a portfolio piece.

**Who it's for:** Hiring managers, peers, and anyone curious about how Shane thinks about workplace architecture and context engineering.

**What it is NOT:** A production SaaS app. No user accounts, no database, no enterprise infrastructure. A polished portfolio project demonstrating context engineering thinking.

**Critical clarification:** The HTML file in `OLD_prototype/` is a **static infographic**, not the interactive SPA described in the spec. There is no extractable source code. The infographic provides only chart data values and the CSS color palette. All interactive features must be **built from the spec**, not extracted from a prototype.

## Part 2: Tech Stack Decision

### 2.1 -- Core Stack: Vanilla HTML/CSS/JS + Vite

| Tool | Purpose | Why This Choice |
|---|---|---|
| **Vite** | Build tool & dev server | Fast, minimal config, ES module support |
| **Tailwind CSS** | Styling (via npm) | Utility-first, matches spec design system |
| **Chart.js** | Charts (radar, bubble) | Covers all chart needs, mature library |
| **DOMPurify** | HTML sanitization | Security-critical; replaces custom sanitize for innerHTML contexts (SEC-01) |
| **Vanilla JavaScript** | Application logic | No framework -- organized JS files |

### 2.2 -- Libraries Removed from Original Concept

| Library | Why Removed |
|---|---|
| **Plotly.js** | ~3MB overhead; Chart.js covers everything |
| **marked.js** | Deferred to Phase 5 (AI features); must pair with DOMPurify (SEC-08) |

### 2.3 -- Hosting: GitHub Pages

- Free for public repos
- Custom domain support (e.g., `modeler.smslo.ai` later)
- If serverless functions are needed (Phase 5), migrate to Vercel
- **Note:** All repos under `smslo-ai.github.io` share localStorage origin. Key is prefixed `context-modeler:` to avoid collisions (SEC-05).

### 2.4 -- Reference Material: Infographic HTML

The file `OLD_prototype/Context_Aware_Enterprise_Infographic.html` is a **static infographic** generated by Gemini. It is NOT the interactive SPA. Extractable assets:

| What | Where in Infographic | Use In Project |
|---|---|---|
| Chart data values | Radar chart percentages, bubble chart coordinates | `data/defaults.js` (Phase 1) |
| CSS color palette | Tailwind classes and hex values | `style.css` / `tailwind.config.js` (Phase 1) |
| Terminology | Section headings, node names | Spec already captures these |

Everything else -- interactions, state management, view switching, forms -- must be built from `SPEC.md`.

## Part 5: Development Workflow

### The Golden Rule: Never Push Directly to `main`

```
1. Create a branch    >  e.g., feature/heatmap-fix
2. Make your changes  >  edit files in VS Code
3. Commit often       >  in GitHub Desktop
4. Push the branch    >  in GitHub Desktop
5. Open a Pull Request on GitHub
6. Review the PR (read your own diff)
7. Merge the PR (squash merge)
8. GitHub Actions auto-deploys to Pages (Phase 3+)
```

### Branch Naming Convention

| Type | Pattern | Example |
|---|---|---|
| New feature | `feature/short-description` | `feature/data-export` |
| Bug fix | `fix/short-description` | `fix/heatmap-random` |
| Setup/infra | `chore/short-description` | `chore/vite-setup` |
| Accessibility | `a11y/short-description` | `a11y/keyboard-nav` |
| Documentation | `docs/short-description` | `docs/update-readme` |

### Commit Message Convention

Start with a verb. Keep it short.

- `Add input sanitization to prevent XSS`
- `Fix heatmap randomness to make values deterministic`
- `Remove hardcoded API key from client-side code`

## Part 7: AI Architecture Decision (Phase 5 Pre-Requisite)

Before starting Phase 5, choose one of three approaches for the AI features.

| Approach | What | Pros | Cons | Best For |
|----------|------|------|------|----------|
| **A: Pre-Written Responses** | Hand-craft 3-5 high-quality analysis responses for default nodes. Display as static content. Label: "Sample AI analysis." | Zero infrastructure. Ships fast. Demonstrates the thinking. | Not interactive. Cannot analyze custom nodes. Visitors may not realize the depth of the concept. | Roles where domain thinking > engineering demonstration |
| **B: Hybrid** | Pre-written responses for default data + live AI for custom nodes added via Input Studio. Single serverless endpoint, not 5. | Best of both worlds. Default experience is instant. Custom nodes show real AI integration. Reduces API surface from 5 endpoints to 1-2. | Still needs serverless setup, but minimal. | Roles where both thinking AND technical execution matter |
| **C: Full AI Native** | All 5 API patterns live. Every button calls a real API. | Most impressive technical demonstration. Interactive. Differentiating. | Highest complexity. 5 endpoints, rate limiting, Redis, CSRF, cost management. Risk of abandoned Phase 5. | Roles specifically targeting AI/ML engineering or platform work |

**Recommendation: Evaluate with an AI engineer before committing.** Key questions to answer:

- Can 5 endpoints be consolidated into 1-2 with prompt routing?
- What is the cold-start latency for the first AI call? Does it ruin the demo experience?
- Can responses be cached per-node to avoid redundant API calls?
- What is the monthly cost at ~50 portfolio visitors/month?

**Decision gate:** Make this choice after Phase 4D ships and the project has been shared with 5-10 people. Their feedback will clarify whether live AI is a differentiator or over-engineering.
